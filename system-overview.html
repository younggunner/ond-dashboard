Operator Next Door

System Architecture & Developer Walkthrough

A geographic neighbor-matching engine that identifies oil & gas operators who share properties, towns, or office buildings with our existing clients — then enriches their contacts and feeds them into a BDR kanban board for outreach.

01 The Thesis

Oil & gas operators are hyper-local. They know who operates on the lease next to theirs. They eat at the same diners. They share the same field offices.

When a BDR calls and says "I see you're on the Lakeside Field with our client Acme Oil — how's production there?" — that's a fundamentally different conversation than a cold call. The operator thinks: "This person knows my world."

Three Types of Local Connection

Platinum — Same Town + Shared Property

Target and our client literally co-operate on the same oil lease AND are in the same town. Strongest possible name-drop.

Script: "I see you and [Client] both operate on [Property Name]. We handle their production reporting — thought you might want the same visibility."

Gold — Same Small Town

Target is in the same small town (<50k pop) as our client. No property connection needed — in a town of 3,000, everyone knows everyone.

Script: "We work with [Client] there in [Town]. Small world — thought I'd reach out."

Silver — Shared Property Only

Target co-operates on a property with our client, but they're in different cities. Weaker but still credible — they share the same lease.

Script: "I noticed you and [Client] are both on [Property] in [County]. We help them track production there."

The Referral Play

Flip it around: instead of selling TO the target, ask our existing client to make an introduction. "Hey [Client], do you know [Target]? You're both on Lakeside Field." Works best with Platinum and Gold.

02 Data Sources

ShaleXP PRIMARY

Public oil & gas database. We scrape operator pages, property pages, and well data. This is the source of truth for who operates where.

Yields: Operators, properties, wells, production dates, addresses

Colorado ECMC CONTACTS

State regulatory database. We scrape the active operators list + employee directories. Direct employee emails and phones from state filings.

Yields: 1,552 employee records with email/phone

Kansas KCC Connect CONTACTS

Kansas Corporation Commission docket system. Service lists from regulatory filings contain operator contacts.

Yields: 905 contacts, 491 emails

US Census Bureau GEO

ZCTA Gazetteer (zip code centroids for geocoding) + Decennial 2020 population counts (for small-town scoring).

Pipedrive CRM CRM

Our CRM. We pull existing org/contact data, enrich it, push it back. Bidirectional sync.

FullEnrich API ENRICHMENT

Bulk email + phone enrichment. Given a name + company (+ optional domain), returns verified corporate email and mobile phone.

TX Railroad Commission CONTACTS

Texas RRC Oil & Gas Directory — bulk download of all active TX operators with officer names, addresses, and phone numbers. Fuzzy-matched to target_operators by company name.

Yields: 7,714 operators → 3,299 matched to targets

Oklahoma OCC CONTACTS

Oklahoma Corporation Commission active operator list (XLSX). Officer names, addresses, phone numbers for all licensed OK operators. Fuzzy-matched to target_operators.

Yields: 2,626 operators → 1,511 matched to targets

TX Comptroller DISAMBIGUATION

Texas state corporate officer lookup. Used to disambiguate contacts with identical names at the same company.

Million Verifier VALIDATION

Real-time email verification. Checks deliverability before we put an email into Pipedrive. Accepts "ok" and "catch_all"; rejects "risky" and "invalid".

Domain-Boost Strategy

If we already have a verified corporate email for anyone at a company (e.g., john@acme-oil.com), we extract the domain (acme-oil.com) and send it to FullEnrich along with the name for unenriched contacts at the same company. This gives FullEnrich a pattern-matching signal — instead of guessing, it can try jane@acme-oil.com, j.doe@acme-oil.com, etc.

Domain sources: Pipedrive contacts (1,530 companies) + KCC scrape (+109) + ECMC scrape (+24) = 1,663 companies with known domains.

04 The Scoring Algorithm

Every target-source pair gets a tier and a numeric score. Three phases run in order; each can promote a pair to a higher tier.

PHASE 1: Property-Based Matching

  For each target that shares a property with a source:

    Same town + shared property?
    ├── YES → PLATINUM (base 100)
    │           + tiny town bonus  (pop < 10k:  +10)
    │           + small town bonus (pop < 25k:  +5)
    │
    └── NO  → SILVER (base 70)
              Different city, but same lease


PHASE 2: Same Small Town (Geography Only)

  For each target NOT already matched:

    Same city + state as a source?
    ├── YES, pop < 50k → GOLD (base 80 + town bonus)
    │
    └── NO, or pop ≥ 50k → skip
              Big cities don't count — too many operators


PHASE 3: Address Proximity (stacks on top)

    Same street address, different suite?
    ├── YES → same-building (+25 modifier)
    │         "You're in Suite 200, our client is in Suite 210"
    │
    ├── Within 0.25 miles? → quarter-mile (+15)
    │   (suppressed in cities > 300k pop)
    │
    └── Within 0.5 miles? → half-mile (+10)
        (suppressed in cities > 300k pop)

Why Suppress Proximity in Big Cities?

In Houston (pop 2.3M), being 0.4 miles apart means nothing — that's a different neighborhood, different building, different world. In Elk City, Oklahoma (pop 12k), 0.4 miles means you're probably neighbors. Only same-building proximity counts in dense metros (>300k).

Score Ranges

TierScore RangeWhat It Means
Platinum100–135Same town + shared property + possible proximity boost
Gold80–95Same small town, no property connection
Silver70–95Shared property, different city (+ possible proximity)

05 Dashboard Filters — What Each One Means

The dashboard map and kanban board have a filter panel. Here's exactly what each button does and what data it includes/excludes.

Tier Filters (Row 1)

FilterWhat's IncludedWhat's ExcludedCount
All Every target operator across all tiers. This is the unfiltered universe. Nothing 2,787
Local + Shared (Platinum) Targets that share a property with our client AND are in the same city. The best leads — strongest name-drop possible. Gold-only and Silver-only targets 264

06 Contact Enrichment

The Goal

For every target operator on the kanban board, we want: decision-maker name + verified corporate email + direct phone number. Without these, the BDR can't make the call.

Enrichment Strategies

1. Standard Enrichment

Send first_name + last_name + company_name to FullEnrich. Works best for well-known companies.

Hit rate: ~23%

2. Domain-Boost Enrichment

If we have a corporate email for anyone at the company, extract the domain and include it. FullEnrich can then pattern-match.

Hit rate: Higher (running now on 5,121 contacts)

3. State Regulatory Scrapes

Colorado ECMC + Kansas KCC provide email/phone directly from state filings. No enrichment API needed — it's public record.

Hit rate: 100% (data is already there)

Filters Applied Before Enrichment

  Contact in pipedrive_contacts
    │
    ├── Dormant? (production ended 4+ years ago) → SKIP
    ├── Bad name? (contains LLC, Inc, "unknown", etc.) → SKIP
    ├── Duplicate? (same first+last at same company) → SKIP
    ├── Org has 4+ contacts AND title is junior? → SKIP
    │     (receptionist, pumper, roustabout, clerk)
    │
    └── Passes all filters → ENRICH

07 The Dashboard

A single-page static HTML file served via GitHub Pages. No backend server — all data is embedded at generation time, with live Supabase sync for kanban state.

Two Views in One

Map View (Top)

Leaflet.js dark map with clustered markers. Each marker = one target operator. Color = tier (purple = platinum, blue = gold, gray = silver). Click to see detail panel with source connections, shared properties, production data.

Kanban Board (Bottom)

5-stage drag-and-drop board. Cards show operator name, city, well count, production, and contact disposition dots. Drag to move through sales stages.

Kanban Stages

  NOT CONTACTED  →  IN PLAY  →  COFFEE DATE BOOKED  →  TRASH TO CASH  →  DEAD FISH
     (2,590)          (0)            (0)                    (0)              (0)

  NOT CONTACTED:  Fresh leads, untouched by BDR
  IN PLAY:        Active outreach — BDR has made contact
  COFFEE DATE:    Meeting booked with the operator
  TRASH TO CASH:  Uses competitor software — revisit later
  DEAD FISH:      Two reasons only — "Not Interested" or "Bad Data"

Per-Contact Activity Logging

Each contact on a card has an activity log. BDR clicks a chip (Voicemail, Bad #, Interested, Meeting) and it logs with a timestamp. Smart prompts: clicking "Interested" offers to move the card to In Play; "Meeting" offers Coffee Date Booked.

Dashboard Features

  • Company-level notes — Yellow sticky-note style, timestamped
  • Contact-level notes — Per-person notes with timestamps
  • System intel — Track which competitor software the target uses (ShaleXP, ScoutFDC, etc.)
  • Edit mode — Toggle to reveal delete buttons for intel, activity entries
  • Time-in-stage — Shows how long each card has been in its current stage (2d, 8h, etc.)
  • Phone formatting — All US numbers displayed as +1 (XXX) XXX-XXXX
  • Bad number handling — Struck-out (not hidden), reversible with restore button

Kanban Persistence Model

  kanban_stages table — compound key: (target_slug, board_key)

  Board keys:
    ond_kanban_stages     → which stage each card is in
    card_notes             → company-level sticky notes

Key Design Decisions

  • Static HTML dashboard — No web server needed. Python generates the file, GitHub Pages serves it. Supabase handles dynamic state.
  • Dual DB adapter — _use_pg = bool(SUPABASE_DSN). Same SQL works on SQLite (local dev) or Postgres (production). Placeholder swap: ? → %s.
  • Regex-based parsing — No BeautifulSoup dependency. ShaleXP HTML is messy but predictable. Regex cascades are faster and more maintainable for this specific HTML.
  • Zip centroid geocoding — Not street-level accurate, but fast and free. Good enough for "are they in the same town?" scoring. No API key needed.
  • Fire-and-forget Supabase writes — Dashboard writes kanban changes to Supabase asynchronously. If Supabase is down, localStorage keeps working. On next load, Supabase wins.
